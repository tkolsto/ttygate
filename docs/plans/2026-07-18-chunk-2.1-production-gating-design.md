# Chunk 2.1 Production Gating and Direct TLS Design

**Issue:** Refs #8 until every acceptance criterion is verified.

**Scope:** Add production-mode startup validation, a typed future trusted-proxy
contract, and direct rustls serving without changing the M1 identity, Origin,
ticket, target, protocol, or PTY lifecycle boundaries.

## Trust model and alternatives

Production is a separate typed operating mode, not a warning level. A production
configuration must select a real authentication contract and exactly one secure
transport boundary. The development identity provider is never real
authentication: because it provisions terminal authority without an external
credential, it is restricted to loopback in development and rejected in
production even when direct TLS is enabled.

Chunk 2.1 defines the minimum future trusted-proxy contract: a validated identity
header name and a nonempty, canonical list of trusted source CIDRs. It does not
read the header, authenticate a proxy, or start a server with that provider.
A complete trusted-proxy production configuration can pass the combination
validator, but application preparation returns a stable provider-unavailable
error before binding until Chunk 2.2 implements peer verification and identity
propagation. This is preferred over a deny-all running production provider,
which would be safe but operationally misleading, and over implementing proxy
authentication early, which would exceed the chunk.

Direct TLS is fully usable with development mode on loopback and is exercised
through the complete browser authority path. This separates transport delivery
from production identity delivery while proving that TLS itself does not weaken
M1. No HTTP fallback is created. No permissive verifier is added to production
code, and ephemeral test certificates are trusted only by test-local clients.

## Typed configuration and validation matrix

`ServerMode::{Dev, Production}` remains the mode boundary. Server transport is
represented as exactly one of plaintext, direct TLS, or trusted proxy. Direct
TLS contains certificate and private-key paths. Trusted proxy contains source
CIDRs and is paired with `auth.provider = "trusted-proxy"` plus its identity
header. Provider-specific stray and missing fields fail closed.

Combination validation enforces:

- development identity requires development mode and a loopback bind;
- production always rejects development identity;
- plaintext requires an `http` public URL and a loopback bind;
- direct TLS and trusted proxy each require an `https` public URL;
- direct TLS and trusted proxy cannot both be configured;
- trusted-proxy authentication requires a complete trusted-proxy transport
  contract, and the contract requires that provider;
- public production binding requires direct TLS or the complete proxy contract;
- URL credentials, paths, queries, fragments, or a scheme inconsistent with
  the selected transport are rejected;
- direct TLS requires both paths, while either partial shape is rejected.

The table-driven matrix names every rejection and representative valid
development/plaintext, development/direct-TLS, and production/trusted-proxy
combinations. Direct TLS is transport rather than authentication, so no
production direct-TLS combination is runnable until a future real provider is
implemented. A configuration may be combination-valid yet still fail
application preparation because Chunk 2.2 has not implemented the selected
production provider. That distinction is explicit in types, errors, tests, and
documentation.

## Certificate and key handling

Direct TLS paths are literal absolute paths. Before reading either file, ttygate
uses symlink metadata and rejects symlinks, non-regular files, empty files, and
files above a conservative fixed bound. On Unix the private key must have no
group or other permission bits; certificate files may be publicly readable.
The files are read only during startup preparation.

rustls configuration construction parses a nonempty PEM certificate chain and
one supported private key, rejects trailing or ambiguous key material, and
verifies that the leaf certificate and private key match. Missing, unreadable,
unsafe, malformed, unsupported, and mismatched inputs map to a closed startup
error taxonomy. Diagnostics identify the logical field and remedy category but
never include PEM contents, private-key material, credentials, parser source
text, or configured filesystem paths. Underlying I/O and rustls errors remain
sources for internal debugging but are not formatted into operator-facing
messages.

`axum-server` with rustls serves the existing axum router over one listener.
The same secure listener therefore carries the frontend, `/healthz`, identity
and session HTTP APIs, and `/api/ws`. There is no parallel plaintext listener.
The TLS integration client installs only its ephemeral CA certificate and uses
normal hostname verification, so a wrong trust root or hostname fails the
handshake rather than being bypassed.

## Startup ordering and error taxonomy

Startup is split into explicit phases:

1. parse the closed TOML schema;
2. validate mode, authentication, bind, transport, URL, and proxy combinations;
3. inspect and load direct-TLS material when selected;
4. construct the existing `OriginPolicy`, authentication provider, allowlist,
   ticket store, and session manager;
5. bind the configured listener;
6. serve either plaintext or TLS, never both.

An injected startup seam records calls to TLS loading, application construction,
and listener binding. Every invalid production case must return before all
later phases; malformed TLS must return before application construction and
binding. Stable public errors distinguish configuration combination, TLS
certificate, TLS private key, TLS key mismatch, unavailable authentication,
application construction, and listener/serve failure without reflecting input.

Shutdown continues to be owned by the existing session manager and PTY
supervisors. Integration fixtures add bounded daemon shutdown, unconditional
temporary directory removal, tracked PTY process-group cleanup, and post-test
absence checks. TLS handshake failure must create no application session or PTY.

## HTTPS, WSS, and retained M1 authority

`public_url` remains the sole exact browser Origin. Under direct TLS it must be
HTTPS, so the frontend derives WSS from the same origin and fixed `/api/ws`
path. Session cookies retain `Secure`, `HttpOnly`, `SameSite=Strict`, and
`Path=/`. TLS configuration does not introduce forwarded-host, forwarded-proto,
identity-header, query, fragment, credential, or subprotocol authority.

The real-TLS integration suite obtains an authenticated development cookie over
HTTPS, lists/chooses only configured presentation metadata, creates a ticket
with the exact configured HTTPS Origin, and redeems it as the first complete WSS
message before reaching a real PTY. It also proves:

- `/` and `/healthz` are served over HTTPS;
- plaintext HTTP is not silently accepted on the TLS port;
- wrong-Origin HTTPS session creation is rejected;
- wrong-Origin WSS upgrade is rejected;
- cookies preserve every required attribute;
- malformed certificates, malformed keys, mismatched pairs, missing files, and
  untrusted/invalid TLS handshakes fail closed;
- query/subprotocol, target, ticket, protocol, and PTY cleanup controls remain
  unchanged.

Focused TLS startup/shutdown, handshake-failure, and cleanup tests run at least
20 times. Temporary certificates, keys, configs, daemons, cookies, tickets, and
PTY markers remain test-local and are removed on success and failure.

## Documentation boundary

README, roadmap, rewrite-plan status, configuration documentation, public-doc
assertions, and threat model will claim only combination gating and direct TLS
that the tests prove. They will explicitly state that trusted-proxy
authentication is still Chunk 2.2 and that production cannot serve until a real
provider exists. Self-signed certificates appear only as ephemeral test
fixtures and are never described as production-safe.

Rate limiting, audit persistence, SSH execution, recording, reconnect,
deployment packaging, proxy examples, and release hardening remain future work.
Changelog-style status references issue #8. `Closes #8` is used only if every
issue and brief acceptance criterion is satisfied; otherwise all commits and
the draft pull request use `Refs #8`.
