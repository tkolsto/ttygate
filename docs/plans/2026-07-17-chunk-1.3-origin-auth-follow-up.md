# Chunk 1.3 Origin and Authentication Boundary Follow-up Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:test-driven-development to implement this plan task-by-task.

**Goal:** Close the post-merge Chunk 1.3 security gaps by making static delivery side-effect-free, moving development identity establishment behind strict Origin authorization, and giving replacement authentication providers bounded request context.

**Architecture:** Static assets remain navigation-compatible and may be fetched without an `Origin`, but they never create or mutate identity state. A new authority-bearing `POST /api/identity` route applies the same strict `OriginPolicy` used by session creation and asks the configured `AuthProvider` to establish a cookie-backed identity from an `AuthContext` containing request headers and optional peer address. The bundled frontend performs this same-origin POST during bootstrap. `POST /api/sessions` revalidates the request context with the opaque cookie before issuing the existing atomic single-use tickets unchanged.

**Tech Stack:** Rust 2024, axum 0.8, Tokio 1, Tower, HTTP headers/extensions, existing `OriginPolicy`, `DevAuthProvider`, and router integration-test harness.

**Issue reference:** Refs #6. This follow-up does not implement WebSocket upgrades, ticket redemption messages, PTYs, SSH, production auth, or trusted-proxy policy.

---

## Critical design review

- **Cross-site behavior:** Browsers commonly omit `Origin` on cross-site subresource GETs. Therefore static GETs cannot safely provision bounded server-side sessions. They remain side-effect-free, while every state-changing browser endpoint (`POST /api/identity`, `POST /api/sessions`, and later WebSocket upgrades) requires exactly one allowed Origin.
- **Replacement boundary:** `AuthContext` exposes read-only request headers and an optional socket peer address on both identity establishment and cookie authentication. The development provider ignores them; Chunk 2.2 can validate configured identity headers only after checking the peer against trusted CIDRs or a constrained local listener on every authority request. No proxy header is trusted in this chunk.
- **Cookie behavior:** Identity establishment returns `204 No Content` and only a `Set-Cookie` header. The cookie remains fixed-size, opaque, `Secure`, `HttpOnly`, `SameSite=Strict`, and `Path=/`. Static responses never set or rotate it.
- **Resource behavior:** Header storage remains axum/http bounded by the incoming request implementation; the provider receives borrowed context and performs no request-controlled allocation. Existing cookie-header and session-map caps remain unchanged.
- **Chunk boundaries:** Ticket issue/redeem, protocol codecs, targets, listener startup, and terminal/session behavior remain unchanged. `OriginPolicy::validate_header_values` remains directly reusable by Chunk 1.5.
- **Bind-default note:** This patch does not make `server.bind` optional because Chunk 1.1 deliberately defines a strict required configuration schema. The documented development configuration value remains `127.0.0.1:7681`; changing config defaults requires a separate contract decision.

## Final acceptance audit

- Static `/`, `/app.js`, and `/app.css` responses contain no identity side effect; supplied hostile Origins remain rejected.
- `POST /api/identity` and `POST /api/sessions` share strict Origin middleware and reject missing, duplicate, malformed, or disallowed values before handlers.
- Development identity remains stored behind a fixed-size opaque `Secure; HttpOnly; SameSite=Strict; Path=/` cookie.
- `AuthContext` supplies borrowed headers and the actual optional socket peer address during both establishment and cookie authentication. Chunk 2.2 can validate trusted proxy headers against peer policy without changing the session/ticket interfaces; this chunk trusts no proxy header.
- The bundled frontend automatically performs the protected same-origin identity POST and exposes only a stable safe bootstrap failure.
- Ticket issuance, redemption, TTL, concurrency, target binding, and safe errors are unchanged and remain sufficient for Chunk 1.5.
- No PTY, SSH, WebSocket route/bridge, production auth, TLS, audit subsystem, or protocol message/codec behavior was added.
