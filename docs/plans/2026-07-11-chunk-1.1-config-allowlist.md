# Chunk 1.1 — Config and Target Allowlist Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add fail-closed TOML configuration loading, validated public domain types, and opaque target-name allowlist resolution for Chunk 1.1 and the dependent GitHub issues #5, #6, and #11.

**Architecture:** Keep file I/O in `config::load`, TOML deserialization in a private schema layer, and semantic validation/conversion into public immutable domain types in `config`. Build a `TargetAllowlist` once from validated targets so request-side code can only resolve an opaque name to an administrator-owned `Target`; no process construction or execution belongs in this chunk.

**Tech Stack:** Rust 2024; `serde` derive; `toml`; `thiserror`; `tempfile` for file-edge tests; standard-library `SocketAddr`, `PathBuf`, `NonZeroU16`, and `Duration`.

---

## Contract and security invariants

- Work only in `.worktrees/chunk-1.1-config-allowlist` on `feat/chunk-1.1-config-allowlist`, based on synchronized `main`; never create or use `master`.
- Public library modules produce `Config`, `ServerConfig`, `AuthConfig`, `AuditConfig`, `Limits`, `Target`, `PtyTarget`, `SshTarget`, `SshUserPolicy`, and `TargetAllowlist` for Chunks 1.3, 1.4, and 3.1.
- Parsing, validation, and file I/O remain separately callable and separately tested.
- Every serde schema object uses `deny_unknown_fields`; unknown target types and user-policy variants fail closed.
- Paths are preserved byte-for-byte as `PathBuf` values after rejecting empty/NUL-bearing values and expansion syntax (`~` prefix, `$`, glob metacharacters, backticks, or shell brace syntax). No canonicalization, joining, environment lookup, shell parsing, or existence check occurs.
- PTY configuration is an array whose first item becomes the executable and whose remaining items become argv. There is no command-string API.
- SSH configuration contains only host, nonzero port, literal known-hosts path, read-only flag, and one typed user policy: `same-as-auth-user`; `fixed` plus `user`; or `mapping` plus a non-empty `user_mapping` table. Policy-specific stray/missing fields fail validation.
- Target names and mapped usernames are non-empty, bounded, printable ASCII identifiers without whitespace or option-like leading `-`; target names accept only ASCII alphanumeric characters plus `.`, `_`, and `-`.
- Limits use positive bounded values. Per-user sessions cannot exceed global sessions; idle timeout cannot exceed absolute lifetime.
- Errors distinguish file read, TOML syntax/schema parsing, semantic validation with an exact field path, duplicate targets/mappings, and unknown allowlist lookup. Error displays never include configuration contents or secret values.
- No axum, Tokio, WebSocket, PTY, SSH spawning, frontend behavior, or Chunk 2.1 production-mode combination gating is implemented.

## Critical pre-execution review

- **Roadmap alignment:** The public types directly satisfy Chunk 1.1’s Produces contract. Lookup is isolated from TOML so Chunk 1.3 can authorize opaque names and Chunk 1.4 can consume immutable PTY targets and limits without parsing configuration again.
- **Configuration-shape alignment:** The complete example remains valid. The plan adds only the minimum fields needed to represent the rewrite plan’s explicitly required fixed and mapping SSH policies; no SSH process behavior is pulled forward.
- **Security alignment:** Private raw types prevent partially validated configuration escaping. Strict unknown-field rejection catches typos; command arrays prevent shell-string ambiguity; path rules reject expansion syntax; policy variants reject stray fields; allowlist lookup accepts only a name.
- **Error-safety alignment:** TOML diagnostics and semantic field paths are actionable, while displays exclude source text and field values. File errors may identify the operator-supplied config path but never config contents.
- **Limits alignment:** Positive, bounded `Duration` and session-count values are ready for Chunk 1.4. Contradictory per-user/global and idle/absolute values fail at startup.
- **Scope control:** Server fields are parsed and typed but unsafe production combinations remain intentionally accepted until Chunk 2.1. Audit settings are configuration only; no logging or recording is implemented.
- **Dependency alignment:** Serde/TOML/thiserror are parser/error dependencies only; tempfile remains dev-only. Accepted runtime choices from ADRs 0002–0004 are consumed as constraints but their dependencies are not added.
- **Architecture question check:** The only newly specified configuration detail is the explicit representation of fixed/mapping SSH policies. Tests will make that contract stable for Chunk 3.1; it does not block Chunks 1.3 or 1.4.
