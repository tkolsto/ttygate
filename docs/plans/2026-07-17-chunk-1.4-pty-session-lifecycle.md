# Chunk 1.4 PTY Session Lifecycle Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Implement Roadmap Chunk 1.4 and GitHub issue #5: allowlisted PTY execution, bounded async session I/O, race-safe limits and timeouts, process-group teardown/reaping, and privacy-safe lifecycle events.

**Architecture:** `SessionManager` reserves global and per-identity capacity before asking a private `pty_backend` module to spawn an existing typed `PtyTarget`. A single session supervisor owns the child and process-group lifecycle while bounded input/output tasks expose a ttygate-owned `Session` API; cancellation uses an independent watch channel so it remains selectable even when I/O channels are full. The public API exposes only typed session state, safe errors, protocol-compatible resize/outcome values, and bounded lifecycle events—never `pty-process`, Tokio child, raw PTY, executable, or argv handles.

**Tech Stack:** Rust 2024, Tokio bounded channels/time/process/I/O, `pty-process` 0.5.3 with `async`, `nix` 0.30 process/signal APIs, existing ttygate `Config`/`Limits`/`PtyTarget`/`Identity`/`Resize`/`ExitStatus` types.

**Issue:** Refs #5. Use `Closes #5` in the eventual PR only after every acceptance criterion and verification item below passes.

## Settled design

### Ownership and module boundaries

- Add `crates/ttygated/src/pty_backend.rs`. It is crate-private and is the only production module allowed to name `pty_process::{Pty, Pts, Command, OwnedReadPty, OwnedWritePty}` or `tokio::process::Child`.
- Add `crates/ttygated/src/session.rs`. It owns public `SessionManager`, `Session`, state/outcome/reason/event/error types, concurrency reservations, timers, bounded channels, and supervisor behavior.
- Inject a crate-private backend trait into `SessionManager`; production uses `PtyProcessBackend`, while unit tests use bounded fakes. The trait returns only ttygate-owned crate-private wrappers, never dependency types.
- Keep `main.rs` unchanged. Export the session API from `lib.rs`; do not wire an HTTP or WebSocket route in Chunk 1.4.
- `SessionManager::start(identity, PtyTarget, initial_size)` accepts a typed target already resolved from `TargetAllowlist`. There is deliberately no string target name, executable, argv, environment, shell fragment, SSH target, or browser option parameter.

### State and events

- Internal state transitions are exactly `Created -> Running -> Closed`; all other transitions return a typed `InvalidTransition` without mutation. Closing an already closed state returns the original terminal snapshot.
- A `Session` returned by `start` is already running, but its bounded event receiver begins with `Created` and `Running`. The supervisor appends exactly one `Closed` event. Event capacity is three, the maximum event count, so no lifecycle event is silently lost and an absent audit consumer cannot block teardown.
- Each event contains identity, configured target name, `SystemTime`, transition, and only the close reason/outcome fields appropriate to that transition. It has no terminal byte field and no ticket, cookie, command, argv, path, or raw error field.

### Bounded I/O and activity

- Input and output payloads are capped with existing `protocol::MAX_BINARY_BYTES`. Invalid oversize input fails with a curated typed error before it reaches the backend.
- Input commands and output chunks use explicitly bounded Tokio `mpsc` channels. The output reader awaits capacity, which pauses PTY reads and applies kernel/PTY backpressure. Its send is selected against the independent cancellation watch channel.
- PTY writes and flushes are also selected against cancellation, so a full terminal input path cannot pin teardown.
- `read_only` is checked both at the public write boundary and inside the worker before any PTY write. Read-only input returns a typed `ReadOnly` result and does not count as activity.
- A successful non-empty PTY input write, successful non-empty PTY output enqueue, or successful resize is activity. Empty I/O and rejected/dropped operations are not. Document this definition in `docs/ttygate-rewrite-plan.md`.
- `Session::read`, `write`, `resize`, and `close` are async. `Resize::new` remains the only dimension validator; the backend receives only a validated `Resize`.

### Timeouts, limits, and teardown

- `SessionManager` protects global and per-identity counters with one mutex, making the check-and-reserve operation atomic. An RAII reservation is acquired before spawn and moved into the supervisor; it releases on spawn failure, normal exit, every close path, task cancellation, or panic unwind.
- Tokio time is the timeout seam. Unit/integration tests use `tokio::time::pause/advance`; lifecycle wall timestamps use a small injected clock trait.
- Idle and absolute deadlines are independent. Activity resets only the idle deadline. Absolute lifetime is anchored when the reservation/session creation begins.
- `pty-process::Command` constructs the executable and argv exclusively from `PtyTarget`, makes the child a session/process-group leader, and attaches its controlling PTY. The backend retains the numeric process-group identity.
- On every close path the supervisor first broadcasts cancellation, stops accepting I/O, sends SIGHUP to the entire process group, waits a short bounded grace period, sends SIGKILL to the entire process group if the leader/group remains, awaits `Child::wait`, and records a portable outcome. A natural leader exit still triggers group cleanup so descendants cannot survive.
- `Drop` on `Session` sends cancellation synchronously. `Session::close` is idempotent and asynchronously waits for the same final closed snapshot; kill-on-drop is defense in depth only.
- The supervisor join handle is private and cannot be aborted by a caller. Cancellation of the future request/WebSocket task drops its `Session`, which requests orderly supervisor teardown. Cancellation/panic of a reader or writer worker is observed by the supervisor and enters the same teardown routine.
- Grace periods and worker joins are bounded. After SIGKILL, the supervisor retains the child and must complete `Child::wait`; it must not time out and drop the child because that would violate the stronger mandatory-reap invariant.

```rust

pub enum TimeoutKind { Idle, Absolute }

    ChildExited,
    Explicit,
    HandleDropped,

pub enum LifecycleTransition {
    Created,
    Running,
    Closed { reason: SessionCloseReason, outcome: Option<ChildOutcome> },
}
pub struct LifecycleEvent {
    pub identity: Identity,
    pub target: String,
    pub at: SystemTime,
    pub transition: LifecycleTransition,
}
```

**Step 3: Implement one teardown routine**

The routine takes sole ownership of the child and PGID. Treat `ESRCH` as already gone; curate every other error. Do not use `Child::kill` as the primary mechanism because it targets only the leader. Do not rely on kill-on-drop.
