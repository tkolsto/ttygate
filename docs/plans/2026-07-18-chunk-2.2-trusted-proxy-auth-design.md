# Chunk 2.2 Trusted Reverse-Proxy Authentication Design

**Issue:** Refs #9. Issue #9 remains open because its audit-attribution
acceptance criterion depends on the future Chunk 2.4 audit subsystem.

**Scope:** Implement the smallest complete trusted reverse-proxy authentication
provider without changing the existing Origin, cookie, ticket, target,
WebSocket, concurrency, session-lifecycle, or PTY authority boundaries.

## Selected architecture and alternatives

The selected design adds a `TrustedProxyAuthProvider` behind the existing
`AuthProvider` interface. It owns the same kind of opaque browser-session store
as development authentication, but it obtains the identity differently:

1. every authority-bearing request must have a listener-supplied
   `ConnectInfo<SocketAddr>`;
2. the socket peer IP must belong to one configured canonical trusted CIDR;
3. only cookie establishment reads the configured identity header;
4. a valid identity is stored server-side under a random opaque token and
   returned through the existing secure browser cookie; and
5. later target, ticket, and WebSocket requests validate the trusted socket peer
   and cookie, but do not reread or replace identity from proxy headers.

This design centralizes peer and header trust in one provider while preserving
the already tested M1 authority pipeline. A listener-only source filter was
rejected because it cannot validate the identity-header grammar or express the
cookie transition. Middleware that copies a header-derived identity into
request extensions was rejected because it would create a second identity path
beside the cookie provider and spread security decisions across handlers.

## Peer provenance and CIDR semantics

The authoritative address is the remote `SocketAddr` produced by the actual
TCP listener and inserted by
`into_make_service_with_connect_info::<SocketAddr>()`. `AuthContext` borrows
that value from the request extension. Absence of listener-provided peer
information fails closed. No address is parsed from `Forwarded`,
`X-Forwarded-For`, `X-Real-IP`, `Host`, query parameters, cookies, WebSocket
subprotocols, or any other client-controlled field.

Configured sources remain the canonical, nonempty, de-duplicated `IpNet` values
validated in Chunk 2.1. Matching uses the peer IP only. IPv4 networks match only
IPv4 peers and IPv6 networks match only IPv6 peers. An IPv4-mapped IPv6 peer is
therefore treated as IPv6 and is never silently converted to IPv4; operators
must configure an explicit IPv6 CIDR if their listener reports mapped
addresses. Network, broadcast, first-outside, last-inside, multiple-CIDR,
loopback, IPv4, and IPv6 boundaries are covered by tests.

Trusted-proxy transport means TLS is terminated by the proxy and the backend
listener is plaintext inside the protected proxy-to-daemon boundary. Direct TLS
remains a separate mutually exclusive transport. A loopback CIDR is supported,
but loopback does not identify a process: operators must prevent other local
processes from reaching the daemon, for example with a dedicated network
namespace, host firewall/packet-filter owner rules, or equivalent OS isolation.
The daemon cannot make a shared loopback port safe by header validation alone.

## Identity-header contract

The provider reads only the configured `auth.identity_header` and requires
exactly one field value. HTTP field-line optional whitespace is framing removed
by the HTTP parser before the provider sees the semantic field value. The proxy
must reject or normalize ambiguous upstream surrounding whitespace and inject a
canonical field line. The semantic value seen by the provider must:

- be valid UTF-8;
- contain 1 through 128 bytes inclusive;
- contain no Unicode whitespace;
- contain no Unicode control character; and
- be accepted byte-for-byte as the canonical, case-sensitive identity.

No trimming, case folding, Unicode normalization, percent decoding, delimiter
splitting, or other lossy transformation of the semantic value is permitted.
Thus empty, whitespace-only, semantic leading/trailing whitespace, embedded
spaces or tabs, oversized, non-UTF-8, and control-containing values fail closed.
Raw field-line optional whitespace may parse to an otherwise valid semantic
value and is therefore covered by the proxy's canonical-injection obligation.
Non-ASCII UTF-8 is accepted when it satisfies the same byte bound and character
rules, and canonically distinct Unicode sequences remain distinct identities.

Duplicate header fields are rejected even if their values are equal. Comma
joining is not performed. Header validation errors are typed internally but
map to stable curated responses that never include the supplied value, peer
text, configured CIDR/header text, cookie, ticket, or parser error.

## Browser cookie, fixation, and authority propagation

`POST /api/identity` retains exact Origin enforcement. With no valid ttygate
cookie, the provider first validates the peer, then the single configured
identity header, creates an opaque random server-side session, and emits the
existing `Secure; HttpOnly; SameSite=Strict; Path=/` cookie. Failure creates no
cookie or terminal authority.

When a valid ttygate cookie already exists, the provider validates the peer and
cookie and returns success without rereading an identity header. A changed,
missing, or hostile identity header therefore cannot silently replace an
existing browser identity. Unknown or malformed cookies do not become valid
through client input; establishment still requires a trusted peer and a valid
single identity header.

The authenticated `Identity` returned from the cookie follows the existing
path unchanged: target listing, `POST /api/sessions`, ticket issuance, ticket
identity binding, WebSocket cookie authentication, first-message ticket
redemption, per-identity concurrency accounting, lifecycle events, and PTY
session ownership. A ticket issued for one identity remains unredeemable by
another identity and is not consumed by the failed attempt.

## Startup, errors, and cleanup

`AppState::from_config` constructs the provider only when
`auth.provider = "trusted-proxy"` is paired with
`server.trusted_proxy.trusted_sources`. Provider construction happens before
binding. An injected construction failure reaches no listener. Trusted-proxy
transport then serves the existing router over the protected backend listener;
there is no public plaintext fallback and no direct-TLS/proxy combination.

Authentication failures use a provider-neutral public response such as
`identity-unavailable` with static text. No failure starts a session or PTY.
Real integration fixtures use controlled loopback peers and the real listener
to prove peer provenance; they do not inject a pretrusted identity. Fixtures
reserve ports through handoff, use bounded shutdown, track PTY process groups,
clean temporary state on success and panic, and prove failed authentication
leaves no daemon task or child behind.

## Documentation and deferred work

README, roadmap, rewrite plan, threat model, public-document assertions, and
retained evidence will describe only tested behavior. Operators must:

- authenticate the user at the proxy;
- remove every client-supplied instance of the configured identity header;
- inject exactly one value satisfying ttygate's identity grammar;
- make the daemon reachable only from configured trusted sources; and
- keep direct public access to the backend blocked.

Proxy compromise or misconfiguration can impersonate any identity and remains
a documented residual risk. Rate limiting, persistent audit output, SSH
execution, recording, reconnect, packaging, deployment examples, release
automation, and release hardening remain future roadmap work.
