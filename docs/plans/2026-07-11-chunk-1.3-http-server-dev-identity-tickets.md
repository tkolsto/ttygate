# Chunk 1.3 HTTP Server, Development Identity, and Tickets Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:test-driven-development to implement this plan task-by-task.

**Goal:** Deliver the Chunk 1.3 localhost HTTP foundation, reusable browser Origin policy, replaceable development identity provider, and bounded atomic single-use session tickets while advancing (not closing) issue #6.

**Architecture:** `server` constructs an axum `Router` from typed application state and remains separate from listener startup. A reusable `OriginPolicy` validates the exact configured public origin before browser-facing handlers run; an `AuthProvider` interface backs opaque cookie sessions; and a mutex-protected `TicketStore` performs bounded issue/redeem/expiry operations atomically over typed `Identity` and existing `Target` values. Static assets are embedded at compile time so the daemon serves the existing built frontend without depending on its working directory.

**Tech Stack:** Rust 2024, axum 0.8, Tokio 1, Tower/Tower HTTP, Serde, cryptographically secure random bytes, HTTP cookies, embedded frontend assets, Node/npm frontend toolchain.

**Scope guard:** No PTY, child process, SSH, WebSocket route/upgrade, terminal UI behavior, production authentication, trusted proxy headers, TLS, audit subsystem, or protocol codec changes. Use `Refs #6`; WebSocket acceptance criteria remain Chunk 1.5.

## Design review and alternatives

1. **Origin enforcement:** Prefer a reusable axum middleware/layer that parses one configured origin and rejects duplicate, malformed, or non-exact request Origins before browser handlers. Authority-bearing endpoints such as `/api/sessions` also reject a missing Origin. Static GET navigation permits a missing Origin because ordinary top-level browser navigation and same-origin subresource GETs commonly omit it, while still rejecting any supplied hostile Origin; otherwise the frontend could not be opened normally. Handler-local checks would be easy to omit on Chunk 1.5, while permissive CORS middleware does not express authorization. `/healthz` is operational rather than browser-facing, and the strict policy API remains directly callable by the future WebSocket upgrade route.
2. **Development identity:** Prefer an `AuthProvider` trait with a development implementation and server-side opaque session IDs. A self-contained signed cookie adds key lifecycle and parsing surface not required by Chunk 1.3. Server-side state makes missing/unknown cookies fail safely, allows auto-provisioning on the frontend bootstrap response, and provides a replaceable boundary for Milestone 2.
3. **Tickets:** Prefer a `std::sync::Mutex<HashMap<...>>` around the complete state transition. A concurrent map still needs coordination to guarantee exactly-one redemption, expiry cleanup, and a strict global capacity. The lock contains only bounded in-memory work and no await points. Random fixed-size bytes encoded as a fixed-length URL-safe opaque token avoid attacker-controlled allocations and make malformed rejection cheap.
4. **Static delivery:** Prefer compile-time embedding of the existing frontend source/build artifacts with deterministic content types. Runtime directory serving is working-directory dependent; invoking npm from Cargo build scripts creates undesirable hidden build-time network/tool dependencies. The embedded inputs must be committed source artifacts or generated by the existing explicit frontend build and included through a minimal Rust asset module.

## Security invariants and limits

- Compare canonical scheme/host/effective-port against the configured public URL; never authorize from `Host`, DNS, loopback, or hostname suffixes.
- Require exactly one syntactically valid `Origin` on every authority-bearing browser endpoint. Navigation-compatible static GETs allow absence but validate any supplied Origin. Stable JSON errors contain curated codes/messages only and never echo request data.
- Cookie: `Secure`, `HttpOnly`, `SameSite=Strict`, `Path=/`, fixed bounded opaque value; no identity or ticket appears in a URL or log.
- Session JSON body is capped before buffering and uses a closed schema. Target names are bounded by the existing config validator and resolved only through `TargetAllowlist`.
- Tickets are fixed-length opaque values, approximately ten-second TTL, identity- and typed-target-bound, removed atomically on successful redemption, and retained on wrong-identity redemption so an attacker cannot burn another identity's ticket.
- Ticket capacity and cleanup scan work are bounded. Expired entries are removed during issue/redeem with a fixed maximum store size; capacity failure is typed and safe.

## Critical plan review

- **Roadmap 1.3 vs 1.5:** This plan deliberately provides redemption and reusable Origin primitives but no WebSocket route, first-message message type, codec change, or terminal bridge. Issue #6 stays open.
- **Rewrite plan:** Identity is real cookie-backed state; ticket is short-lived, single-use, bound to identity and an existing typed allowlisted target, and absent from URLs/logging.
- **Threat model:** Exact Origin authorization covers cross-site, suffix-confusion, port-confusion, and DNS-rebinding abuse without relying on loopback or `Host`. Negative tests dominate each browser boundary.
- **Protocol:** No authentication message is added to the terminal codec. Chunk 1.5 can redeem before handing any message to the existing codec, preserving its scope.
- **Resource safety:** Fixed token/cookie lengths, bounded request bodies, closed schemas, capped store entries, and bounded cleanup prevent request-controlled unbounded allocation/work. The store lock contains no async or external work.
- **Config reuse:** `Config.server`, `AuthConfig`, `TargetAllowlist`, and cloned typed `Target` values are consumed; no competing target/config representations are introduced.
- **Open question resolved:** Wrong-identity redemption does not consume a valid ticket, avoiding an unauthenticated denial primitive; successful redemption remains exactly-once under concurrency.
