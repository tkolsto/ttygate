# Chunk 0.2 — Public Docs Baseline Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Apply superpowers:test-driven-development to executable documentation checks and superpowers:verification-before-completion before each commit and the PR.

**Goal:** Publish an accurate, security-forward documentation baseline and organize the public roadmap into 12 milestone-grouped GitHub issues without overstating the pre-release scaffold's current capabilities.

**Architecture:** Repository documentation is the source of truth: `README.md` gives the safe public entry point, `SECURITY.md` owns private vulnerability reporting and version support, and `docs/threat-model.md` defines the security boundaries and residual risks. GitHub issue forms route ordinary security-sensitive engineering work into structured public reports while directing suspected vulnerabilities away from public disclosure; repository milestones and issues mirror the rewrite plan's phases.

**Tech Stack:** GitHub-flavored Markdown, GitHub issue-form YAML, POSIX shell validation, GitHub CLI/API, existing Rust and npm CI.

---

## Global constraints

- Work only in the project-local worktree on `docs/chunk-0.2-public-docs`, based on `origin/main`; never create `master` or push to `main`.
- Preserve the clean-room rule: do not copy or adapt code or prose from Shell In A Box or its forks.
- Preserve dual licensing as `MIT OR Apache-2.0` and link both license files.
- Describe only behavior present in the current repository as available. Mark planned server, authentication, terminal, SSH, audit, and production features as unavailable until implemented.
- Never place vulnerability details in public issue templates or public issues; route them to GitHub private vulnerability reporting/security advisories.
- Add no `TODO` or `FIXME` comments and no AI attribution.
- Use conventional commits and inspect every test/check result before proceeding.

### Task 3: Publish the full threat model

**Files:**
- Create: `docs/threat-model.md`
- Test: `scripts/check-public-docs.sh`

**Step 1: Write `docs/threat-model.md` from the ttygate rewrite plan**

Cover all of the following, using original prose:

- Purpose, status, security objectives, scope, assumptions, and explicit non-goals.
- Assets: host/daemon authority, authenticated identities and cookies, one-time tickets, terminal data, target/SSH policy and known-hosts data, audit logs, recordings, availability, and configuration/TLS keys.
- Trust boundaries: browser↔service, HTTP API↔WebSocket ticket redemption, reverse proxy↔daemon, daemon↔PTY/SSH children, authenticated user↔shared daemon OS account, terminal output↔browser renderer, daemon↔logs/recordings, and config/admin↔runtime.
- Attacker capabilities from the rewrite plan: unauthenticated remote clients, malicious sites in a legitimate user's browser, malicious authenticated users, compromised/misconfigured proxies, hostile terminal output, local unprivileged users, network attackers under bad TLS boundaries, resource exhaustion, and unsafe administrator choices.
- Threats and planned mitigations in a table: cross-site WebSocket hijacking/CSRF/DNS rebinding; ticket theft/replay; auth/header spoofing; target/argument injection; session crossover; privilege escalation; terminal-output attacks; SSH MITM; transport interception; secret leakage; resource exhaustion/orphan children; unsafe configuration; dependency/supply-chain risk.
- Minimum controls in every mode, production-only controls, audit/recording rules, dangerous anti-features, and validation strategy.
- Residual risks: shared OS-user model, recordings and terminal output containing secrets, authenticated command authority, compromised proxies, endpoint/host compromise, denial of service within configured limits, audit metadata sensitivity, and pre-release immaturity.
- A maintenance rule requiring updates when trust boundaries, auth, execution, transport, logging/recording, or deployment assumptions change.

Clearly distinguish planned mitigations from controls already implemented in Chunk 0.1.
