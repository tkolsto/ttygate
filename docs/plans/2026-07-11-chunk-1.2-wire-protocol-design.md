# Chunk 1.2 Wire Protocol Design

## Scope and goals

Chunk 1.2 defines a small, versioned WebSocket payload contract and pure codecs for Rust and TypeScript. It does not open sockets, manage browser connections, establish sessions, redeem tickets, authenticate users, or interpret terminal bytes. Ticket redemption happens before this protocol begins carrying terminal traffic.

The contract must be safe to parse at an attacker-controlled boundary, useful to the PTY/session, bridge, and frontend chunks, and directly fuzzable later. The specification is authoritative; a shared fixture file makes implementation drift fail in both languages.

## Considered approaches

1. **Strict hand-written codecs plus shared fixtures (selected).** Rust and TypeScript expose native typed APIs and validate the same committed wire cases. This keeps runtime code small, produces ergonomic domain types, and catches representation drift without introducing a generator.
2. **JSON Schema plus generated types.** This centralizes structure but does not naturally express WebSocket directionality, binary frames, safe error taxonomy, or all integer constraints. It also adds build tooling before the project needs it.
3. **A single bidirectional message union.** This is superficially simpler but makes it easy for later bridge code to accept server-only messages from a client. Separate client/server decoders make invalid direction unrepresentable at the public API boundary.

## Protocol shape

Protocol version `1` is exact-match: unsupported versions fail closed. Binary frames are opaque terminal bytes. Their meaning depends on direction: client-to-server is terminal input; server-to-client is terminal output. Empty binary frames are valid and each frame is limited to 65,536 bytes.

Text frames are UTF-8 JSON objects limited to 4,096 encoded bytes. Every object has exactly `version` and `type` plus variant fields. Client controls are `resize` (`cols`, `rows`, each integer 1–4096) and `close` (no extra fields). Server controls are `exit-status` with a tagged status (`code` 0–255, `signal` 1–127, or `unavailable`), `error` with a stable kebab-case code and safe plain-text message, and `close` with a stable reason. Unknown types, fields, missing fields, malformed JSON, wrong types, unsupported versions, invalid ranges, and wrong-direction messages are rejected.

Errors are typed categories and never contain source payloads. A protocol violation is fatal: the bridge will send a generic protocol error when safe and close with WebSocket policy-violation semantics. Resource-limit errors close immediately. Backpressure is owned by later runtime chunks, which must use bounded queues, pause producers when full, and remain cancellation-selectable.

## Components and testing

Rust adds `ttygated::protocol`, independent of axum, with typed client/server enums, frame enums, pure encode/decode functions, constants, and actionable non-reflecting errors. TypeScript adds `frontend/src/protocol.ts`, independent of WebSocket and xterm.js, with matching discriminated unions and pure functions. Both distinguish binary bytes from JSON text at their APIs.

`protocol/fixtures.json` contains deterministic valid and invalid cases. Both suites consume it, assert exact encodings, and independently add native boundary tests including invalid UTF-8 on the Rust text boundary. Fixture constants are asserted against implementation constants, preventing undocumented changes. Tests never open a socket or start a server.
