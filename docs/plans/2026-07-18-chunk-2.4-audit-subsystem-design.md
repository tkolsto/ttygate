# Chunk 2.4 Audit Subsystem Design

**Issue:** Closes #10.

## Purpose and boundaries

Chunk 2.4 adds the durable security audit control required by the Phase 2 roadmap. The audit log must answer who authenticated, who attempted or opened which configured target, from which listener-reported socket address, when the session started and ended, and why an authority request or session ended. It must do that without becoming a credential store or terminal recording.

The audit subsystem will never accept arbitrary maps, raw errors, request text, headers, cookies, tickets, command arrays, environment values, or terminal bytes. Producers construct closed Rust enums and validated domain values. Target-bearing constructors require a `ResolvedAuditTarget` obtained from the configured allowlist rather than a raw string. Serialization is centralized so future SSH host-key failure work can add a bounded enum variant without creating an independent logger.

This chunk does not implement terminal recording, SSH execution, reconnect, rotation, remote shipping, database storage, packaging, deployment examples, or release automation. The pre-release build remains unsuitable for production deployment.

## Considered approaches

### 1. Reuse the session lifecycle broadcast

The existing bounded broadcast is useful for observers and tests, but it is intentionally lossy when receivers lag and does not cover HTTP or WebSocket denials. A background subscriber also cannot make an authority request fail synchronously when persistence is unavailable. It remains an in-memory observation API, not the durable control.

### 2. Add a bounded asynchronous writer task

A bounded queue could keep filesystem latency off request paths. It would require a backpressure policy, writer-task lifecycle, shutdown draining, panic recovery, and a way to report a later disk failure to the request that already received authority. Dropping records is unacceptable; blocking on a full queue becomes equivalent to synchronous writing with more failure states.

### 3. Serialize and append inline under one mutex

This is the selected design. One process-owned `AuditLog` validates and opens the configured destination during startup, serializes one closed event into one bounded byte vector, appends exactly one newline, and performs the write while holding one mutex. Concurrent producers therefore have one deterministic serialization order and cannot interleave JSON fragments. Authority paths can fail closed before returning a cookie, ticket, or live session when the sink is unavailable.

The trade-off is filesystem latency on security-authority paths. Audit records are small and bounded, and correctness is more important than throughput for the current pre-release scope.

## Versioned event contract

Every line is one JSON object with `schema_version: 1`, a stable `event_type`, and an RFC 3339 UTC timestamp. Exact serialization tests pin field names, enum spellings, omission behavior, escaping, and newline handling.

The initial closed event variants are:

- `authentication-succeeded`: authenticated identity and authoritative listener socket address when available;
- `session-started`: opaque random session identifier, identity, configured target name, authoritative listener socket address when available, and UTC start time;
- `session-ended`: the same session attribution, UTC start/end times, and one stable close reason plus a portable exit outcome when available; and
- `access-denied`: an opaque correlation identifier, authoritative listener socket address when available, optional already-authenticated identity and configured target, plus closed denial category and reason enums.

Denial categories are `authentication`, `rate-limit`, `capacity`, `target`, `ticket`, `origin`, and `host-key`. The host-key category is representable now but emitted only when the SSH backend is implemented. Reasons are bounded stable enums such as `identity-required`, `authentication-rate-limited`, `global-session-limit`, `target-not-found`, `ticket-expired`, and `origin-denied`; no attacker-controlled text is serialized.

Session identifiers and denial correlation identifiers are generated from 128 bits of OS randomness and encoded as unpadded URL-safe base64. They are references, never authorization values. Identity comes only from the configured authentication provider. Target appears only after allowlist resolution. Remote address comes only from axum listener `ConnectInfo`; `Forwarded`, `X-Forwarded-For`, `X-Real-IP`, Host, query parameters, and WebSocket subprotocols are never address or identity authority.

## File and failure contract

`[audit].path` remains the existing literal typed path. No tilde, environment-variable, glob, command, or brace expansion is added. Startup opens that exact path before application construction or listener binding.

On Unix, every path component is checked for symlinks and then opened through
an anchored `openat` directory-descriptor walk with no-follow semantics.
Parent renames cannot redirect the final open. The destination uses
append/create/read-write, no-follow, close-on-exec, and nonblocking flags so a
raced special file cannot block startup; a new file is requested with mode
`0600`. The opened handle is revalidated as a regular file. An existing file is
accepted only when group and other permission bits are absent; ttygate never
chmods an existing file and therefore never silently weakens or rewrites
operator policy. Missing parent directories, directories, symlinks, special
files, incomplete existing JSONL tails, permission failures, and unwritable
destinations fail with stable non-reflecting startup categories.

Each event is serialized before taking the writer mutex. The mutex protects a single append operation and the writer state. A write failure permanently poisons the sink: there is no fallback destination, reopen loop, stderr copy, or silent continuation. Subsequent authentication, ticket, and session authority is denied with a stable service-unavailable response. Sessions that were already running continue under their existing supervised timeouts and cleanup paths; Chunk 2.4 does not kill them solely because later authority encountered audit loss. Their end record may be unavailable after the filesystem failure that caused degradation. Existing complete records remain untouched, while a failed final append may leave an incomplete tail that causes the next startup to fail rather than concatenate corrupt JSON.

Writes reach the kernel through the opened file descriptor before authority is returned. This chunk does not call `fsync` per event, so acknowledged records are not guaranteed to survive sudden power loss or kernel failure. The operator owns the filesystem, capacity monitoring, access controls, retention, backups, deletion, and any external collection.

## Lifecycle integration

HTTP Origin middleware records wrong-Origin authority requests before returning
403. Authentication establishment, HTTP cookie authentication, and WebSocket
upgrade authentication record stable authentication/rate denials; successful
establishment records only the authenticated identity and socket provenance.
Session creation records rate, unknown-target, ticket-capacity, and
global/per-identity capacity denials without reflecting request bodies or
unknown target text. A sink mutex guard linearizes successful ticket insertion
against concurrent permanent audit failure, so no ticket can be issued after
that failure wins the ordering race.

The WebSocket upgrade captures the listener socket address and passes it with
the authenticated identity into ticket redemption and session admission.
Malformed, unknown, expired, wrong-identity, and replayed tickets emit stable
ticket denials without the ticket. Unsupported configured targets,
post-redemption reservation failures, and manager shutdown emit stable denials
without orphaned starts. A valid configured target and reservation transfer
into the session manager.

The session manager generates the opaque session identifier, starts the
allowlisted PTY, writes one `session-started` record, and gives the supervisor
sole ownership of the matching `session-ended` record. Normal exit, explicit
close, WebSocket disconnect/handle drop, protocol violation, internal bridge
failure, idle timeout, absolute timeout, spawn failure, worker/backend failure,
shutdown, cancellation of callers, resistant-child escalation, and unwind
recovery keep the existing process-group teardown and exactly-once close state.
Spawn failure is a denial and never creates a start record. A supervisor unwind
is caught at the task boundary, runs bounded child cleanup, and emits one
internal-failure end when the audit sink remains writable.

The existing in-memory `Created → Running → Closed` lifecycle stream remains compatible for current consumers and tests. Durable audit deliberately exposes one coherent admitted-session start/end pair rather than duplicating internal state transitions.

## Secrecy and verification

Audit event types have no fields capable of containing terminal input/output, request bodies, raw headers, cookies, tickets, authorization values, TLS material, private keys, command arguments, environment contents, WebSocket subprotocols, or raw error strings. Exact JSON and hostile-newline tests prove JSON escaping cannot inject another record.

Integration tests use distinctive sentinels for typed terminal input, terminal output, cookie, ticket, header, query, command argument, and environment data, then scan the complete JSONL file and parsed values. Concurrent-writer tests parse every produced line independently. Restart tests preserve pre-existing complete lines. End-to-end authority tests reconstruct identity, configured target, address, start, and end solely from the audit log across development, direct-TLS, and trusted-proxy modes.

## Residual risks

Audit metadata remains sensitive: identities, targets, socket addresses, timestamps, denial patterns, and outcomes reveal operations even without terminal contents. A local user with filesystem or daemon-user access can read, truncate, replace, or delete logs within OS permissions. Shared daemon OS-user execution is not per-identity isolation. Disk exhaustion can deny new authority and can prevent an active session's final record. Rotation, retention, backup, tamper evidence, remote storage, and availability across host compromise remain operator responsibilities or deferred work.
