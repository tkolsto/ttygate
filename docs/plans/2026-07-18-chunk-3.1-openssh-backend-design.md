# Chunk 3.1 Strict-Host-Key OpenSSH Backend Design

**Issue:** [#11](https://github.com/tkolsto/ttygate/issues/11)

**Scope:** Execute configured SSH targets through a directly spawned system
OpenSSH client while preserving ttygate's existing authority, audit, protocol,
capacity, and PTY cleanup boundaries.

## Decisions

### Credential and executable policy

Each `type = "ssh"` target gains two required literal, administrator-controlled
paths:

- `ssh_executable`, the exact OpenSSH client to execute; and
- `identity_file`, one structurally complete, unencrypted OpenSSH Ed25519
  private-key envelope used only for that target.

This is intentionally narrower than ambient OpenSSH behavior. ttygate does not
search `PATH`, read a user SSH configuration, discover default identities, use
`SSH_AUTH_SOCK`, forward an agent, or accept credentials from the browser. An
SSH target is public-key-only. Password, keyboard-interactive, host-based, and
GSSAPI authentication are disabled. Encrypted identity files are rejected
because ttygate has no credential-prompt channel and must not pass a prompt
through the browser terminal.

Alternatives were considered and rejected:

1. An explicit literal identity file is selected because its authority is
   visible in typed server configuration and can be validated before bind.
2. An SSH agent or default identity discovery is rejected because process
   environment and operator login state would silently change daemon authority.
3. Browser-provided keys, key paths, or passphrases are rejected because they
   enlarge the protocol and secret-handling boundary.

### Filesystem trust boundary and startup validation

Before application construction or listener bind, every configured SSH target
is prepared. Paths remain literal: no tilde, environment-variable, relative-path
canonicalization, or symlink expansion is performed.

The executable must be an absolute, non-symlink regular file, owned by root or
the daemon effective user, executable by the daemon, bounded in size, and not
group/world writable. A bounded `ssh -G` capability probe, invoked directly
with the complete mandatory option vocabulary and a cleared deterministic
environment, must succeed without network access.

Known-hosts and identity files are opened with `O_NOFOLLOW`, which rejects a
symlink only in the final path component. Parent components may still resolve
through symlinks, so the administrator-controlled parent namespace is an
explicit trust boundary rather than something this check proves. Both files
must be absolute regular files owned by the effective daemon user. Known-hosts data
must be nonempty, bounded, newline-complete, NUL-free, and not group/world
writable. It may be group/world readable because it contains public host keys.
The identity must be nonempty, bounded, owner-only, newline-complete, NUL-free,
contain exactly one structurally complete supported private-key envelope, and
be demonstrably unencrypted. The structural parser deliberately does not derive
the public key from the Ed25519 seed or prove cryptographic consistency:
ttygate is a bounded credential-envelope parser, while OpenSSH remains the
credential parser and cryptographic authority. Directories, final-component
symlinks, special files, unsafe permissions, wrong
ownership, empty files, oversized files, incomplete/trailing material, and
raced replacements fail with stable path-free startup errors.

Prepared metadata snapshots include device, inode, owner, mode, size, and
modification/change timestamps. The same checks and snapshot comparison run
immediately before each spawn. OpenSSH necessarily reopens the configured
literal paths, so an attacker controlling the filesystem can still replace a
path after ttygate's final check and before OpenSSH's reopen. This subprocess
TOCTOU limit is documented rather than hidden. Safe operation therefore assumes
the configured files and every parent namespace component (including any
symlink target) are controlled by the daemon administrator and not writable by
SSH users.

### Exact OpenSSH process contract

ttygate executes the validated literal executable directly, never through a
shell. It clears the environment and sets only fixed server constants needed
for deterministic classification and terminal compatibility:
`LANG=C`, `LC_ALL=C`, and `TERM=xterm-256color`.

The argv builder emits every security option exactly once:

```text
-vv
-tt
-E <server-generated private FIFO path>
-F /dev/null
-o StrictHostKeyChecking=yes
-o UserKnownHostsFile=<literal configured path>
-o GlobalKnownHostsFile=/dev/null
-o UpdateHostKeys=no
-o CheckHostIP=yes
-o BatchMode=yes
-o IdentitiesOnly=yes
-o IdentityFile=<literal configured path>
-o IdentityAgent=none
-o AddKeysToAgent=no
-o PreferredAuthentications=publickey
-o PubkeyAuthentication=yes
-o PasswordAuthentication=no
-o KbdInteractiveAuthentication=no
-o ChallengeResponseAuthentication=no
-o HostbasedAuthentication=no
-o GSSAPIAuthentication=no
-o ForwardAgent=no
-o ForwardX11=no
-o ForwardX11Trusted=no
-o ClearAllForwardings=yes
-o PermitLocalCommand=no
-o ProxyCommand=none
-o ProxyJump=none
-o EnableEscapeCommandline=no
-o EscapeChar=none
-o CanonicalizeHostname=no
-o RequestTTY=force
-p <configured port>
-l <resolved configured-policy username>
-- <configured host>
```

`ssh -G` verification will keep only options supported by the repository's
documented OpenSSH compatibility floor. If an option is not supported, startup
fails closed; there is no weaker runtime fallback. The host, port, paths, and
fixed policy values come from validated target configuration. The username
comes only from the configured fixed, same-as-authenticated-user, or mapping
policy. Browser traffic contributes only the previously authorized opaque
target name, then opaque terminal bytes and bounded controls.

Resolved usernames must be valid UTF-8, 1 through 128 bytes, start with an
ASCII alphanumeric character, and otherwise contain only ASCII alphanumerics,
dot, underscore, or hyphen. They may not end in a hyphen, begin with an option
prefix, contain whitespace/control/confusable Unicode, or alter argv structure.
Missing mappings and invalid resolved names fail before spawn.

### Admission, diagnostics, and failure classification

OpenSSH receives a real PTY for stdin/stdout so interactive remote sessions and
resize continue to work. Raw child stderr is a separate pipe consumed only by a
drain-and-discard worker; it is never classified, forwarded, stored, audited, or
logged. Admission classification reads only OpenSSH's `-E` client log through a
server-created FIFO inside a per-session `0700` temporary directory. The FIFO
is `0600`, opened server-side `RDWR|NONBLOCK` before spawn, continuously drained
through readiness-based async I/O, and removed with its private directory when
the backend is dropped. It is not a growing regular log.

The temporary parent namespace and same-daemon-UID boundary remain trusted:
another process with that UID can inspect or interfere with daemon-owned
materials. Task 6 must add the real sshd malicious-banner regression proving a
server-written stderr banner cannot forge admission; no such real-sshd evidence
is claimed by Task 3.

The PTY boundary is not operating-system isolation. Any local PTY target running
as the daemon UID can read daemon-readable SSH identities and other files
permitted to that UID. Deployments that allow less-trusted local commands must
use separate operating-system identities or a stronger sandbox.

A local child spawn is not SSH admission. The registered supervisor owns the
process group and capacity lease while connection setup is pending. The session
becomes Running, starts its idle/absolute deadlines, exposes terminal I/O, and
emits exactly one correlated `session-started` event only after the bounded
client-log classifier observes a complete anchored OpenSSH authentication line
for the exact configured host using `publickey`.

Before that marker, child exit or a classified diagnostic produces exactly one
denial and no session start/end pair:

| Internal class | Frontend code | Audit category/reason |
|---|---|---|
| unknown host key | `ssh-host-key-failed` | `host-key/unknown-host-key` |
| changed/mismatched key | `ssh-host-key-failed` | `host-key/host-key-mismatch` |
| connect/transport failure | `ssh-connection-failed` | `target/ssh-connection-failed` |
| authentication failure | `ssh-authentication-failed` | `authentication/ssh-authentication-failed` |
| username mapping/grammar denial | `ssh-policy-denied` | `target/ssh-user-policy-denied` |
| spawn failure | `session-unavailable` | `target/session-unavailable` |
| bounded unknown setup failure | `ssh-failed` | `target/ssh-failed` |
| setup timeout | `ssh-connection-failed` | `target/ssh-connection-failed` |
| internal/audit failure | `session-unavailable` | `target/audit-unavailable` or stable internal denial |

Classification never returns raw diagnostics. Version- or locale-dependent text
that does not match the bounded supported patterns falls back to `ssh-failed`.
Unknown and changed keys remain distinct audit reasons even though the frontend
uses one safe host-verification state.

After admission, a normal remote exit is a normal child outcome and preserves
the existing portable exit-status/one-start/one-end lifecycle. Local explicit
close, WebSocket drop, idle/absolute timeout, cancellation, shutdown, worker
failure, and unwind use the existing supervised process-group teardown.
Capacity remains held until the local SSH process group is reaped.

### Cancellation and residual remote authority

The connecting child is entered in the same supervisor registry before the
caller waits for admission. This closes the cancellation race: caller drop,
manager shutdown, audit failure, timeout, or unwind signals and reaps the local
process group. The supervisor owns the capacity lease until cleanup completes,
including resistant-child SIGKILL fallback.

Killing and reaping local `ssh` proves local cleanup only. If transport is lost
after remote work begins, ttygate cannot guarantee when or whether sshd and the
remote operating system terminate remote work. No reconnect, attach, remote
command, or remote process-control protocol is added in Chunk 3.1.

### Protocol and frontend

The wire schema does not gain any SSH authority fields. It gains only curated
server error codes already permitted by the closed generic error envelope.
The frontend maps those codes to stable states and fixed copy:

- host identity could not be verified;
- SSH connection could not be established;
- SSH authentication was rejected;
- SSH user policy denied the session; and
- SSH session setup failed safely.

Copy never reflects configured target names, hosts, usernames, paths, argv,
stderr, or request values. Existing terminal input/output, resize, close,
backpressure, timeout, and stale-connection behavior remains unchanged.

## Test strategy

Strict TDD adds named unit, integration, real-browser, and real-sshd tests before
each production slice. Unit tests prove configuration/file rejection, literal
paths, username policy, exact argv/environment, no-shell construction,
diagnostic bounds, and stable mappings. Session/WebSocket tests prove
pre-admission denials have no lifecycle start, admitted sessions complete
exactly once, capacity/cancellation races remain owned, and secrets never enter
audit/frontend data.

The containerized fixture generates all keys and known-host data inside a
private temporary directory, provisions trust from the generated server public
key rather than the connection under test, binds sshd to random loopback
ports, and always removes containers and artifacts. It covers strict success,
empty/unknown and mismatched host keys, refused transport, bad credentials,
remote exit, resize, opaque I/O, every cleanup path, and container-side absence
of lingering session children.

Independent security review covers argument and option injection, ambient
authority, file races, permission/ownership checks, leakage, diagnostic
portability, false lifecycle success, capacity races, process cleanup,
frontend reflection, and documentation truth. Every valid finding first gets a
named failing regression, then the smallest fix, then re-review until no
Critical, Important, or Minor findings remain.

## Deferred scope

Native SSH, TOFU or automatic key learning, agents and agent forwarding,
password/keyboard-interactive authentication, encrypted keys, remote commands,
SCP/SFTP, port forwarding, reconnect, attach, recording, packaging, deployment
examples, and release work remain deferred. Chunk 3.1 does not weaken the
pre-release warning.
