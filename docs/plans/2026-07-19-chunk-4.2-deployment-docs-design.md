# Chunk 4.2 Verified Deployment Documentation Design

## Scope

Chunk 4.2 turns the trusted reverse-proxy contract implemented in Chunk 2.2
into operator-readable Caddy and Nginx deployments and proves both examples
with a disposable end-to-end harness. It also completes the public
documentation assigned to the v0.1 README checklist. It does not weaken
production validation or add release automation, checksums, SBOMs, signing,
published images, artifacts, tags, or a production-readiness claim. Those
remain Chunk 4.3 work.

The examples use an external authentication subrequest because ttygate does
not implement Cloudflare Access, Tailscale authentication, OIDC, or another
identity provider. The synthetic authentication service in the smoke harness
is a test fixture, not an operator component or credential system.

## Alternatives considered

The selected design runs the exact committed Caddy and Nginx configurations
inside one shared disposable Docker harness. The harness gives both proxies
the same ttygate backend, synthetic authentication contract, TLS certificate,
client flow, audit assertions, and cleanup policy. It also makes the complete
Linux proxy lifecycle reproducible from macOS and in required CI.

A host-native harness would more closely resemble a systemd installation, but
would require installing and coordinating two proxy packages, would not run
fully on the macOS development host, and would make backend reachability
isolation harder to prove consistently. Syntax checks plus ttygate's existing
in-process Rust proxy fixture would be faster, but would not prove that the
committed Caddy and Nginx examples themselves preserve the HTTP, WebSocket,
identity, and cleanup contract.

## Deployment security contract

The browser reaches only the TLS-terminating proxy. The ttygate listener uses
plaintext HTTP only inside a private loopback or isolated service network and
is not published as a client endpoint. Its `public_url` is the exact external
HTTPS authority, so HTTP session requests and WSS upgrades enforce the browser
Origin that users actually visit. Secure, HTTP-only, SameSite cookies remain
required.

The proxy first authenticates the request with a dedicated upstream auth
gateway. A missing or rejected authenticated identity never reaches ttygate.
The proxy replaces every client-supplied instance of the configured identity
header with exactly one canonical value obtained from that authentication
decision. It also removes upstream authorization credentials before forwarding
to ttygate. Caddy expresses this with `forward_auth`, copied response headers,
and `header_up`; Nginx uses `auth_request`, `auth_request_set`, and
`proxy_set_header`. Their configurations remain separate because their header
and WebSocket models are not identical.

ttygate independently checks the actual backend socket peer against the exact
configured trusted CIDR before examining the identity header or accepting an
existing cookie. `Forwarded`, `X-Forwarded-For`, `X-Real-IP`, Host, and other
client fields never establish peer or identity authority. The proxy and
ttygate therefore provide complementary controls: header replacement is not a
substitute for backend network isolation, and a trusted CIDR is not a
substitute for proxy authentication.

The same path applies to frontend, health, identity establishment, ticket
issuance, and WebSocket upgrade. A compromised trusted proxy can impersonate
any identity and is an explicit residual risk.

## Operator examples

`packaging/reverse-proxy/` contains:

- a complete `Caddyfile`;
- a complete Nginx configuration;
- a matching production ttygate configuration with conspicuous replacement
  markers for the public authority, trusted proxy CIDR, audit path, and target
  material;
- a deployment guide covering invocation, ownership, permissions, TLS and
  secret handling, WebSockets, health checks, audit persistence, rotation,
  retention, shipping, troubleshooting, and network isolation; and
- Cloudflare Access-style and Tailscale Serve-style identity-header guidance.

The generic Caddy/Nginx examples expect an auth service which returns exactly
one canonical `X-Authenticated-User` response header on success and denies
requests without a valid upstream session. Cloudflare guidance requires either
Cloudflare Tunnel to validate the Access token or independent JWT signature
and claim validation at the origin; trusting the email header by itself is
explicitly prohibited. Tailscale guidance uses `Tailscale-User-Login`, notes
that Serve removes client copies before injection, and keeps ttygate on
loopback. It also documents that Funnel and tagged-device traffic do not
provide that user identity.

The guide links to the primary Caddy, Nginx, Cloudflare, and Tailscale
documentation supporting material operator claims. Test certificates and
synthetic fixture identities are clearly separated from production
certificate, identity-provider, and secret handling.

## Shared executable harness

`scripts/check-deployment-docs.sh` supplies fast platform-independent contract
assertions. It checks required files and documentation, rejects unsafe example
defaults, verifies the external HTTPS/public URL relationship, checks
proxy-specific identity replacement and WebSocket directives, ensures CI uses
immutable proxy image and action revisions, and validates the README checklist.

`scripts/smoke-reverse-proxy.sh <caddy|nginx>` creates one isolated internal
Docker network and runtime directory. It generates a short-lived certificate
for the reserved example authority, renders the conspicuous ttygate
configuration markers with the disposable proxy address, and starts:

1. ttygate with no published port;
2. a bounded synthetic authentication service;
3. exactly one selected proxy using the committed configuration; and
4. a client fixture attached only to the proxy-facing network path.

Before startup, the selected proxy runs its native validation command against
the exact committed configuration and generated certificate paths:
`caddy validate --config` or `nginx -t -c`. The client verifies HTTPS,
`/healthz`, frontend assets, missing-auth denial, spoofed-header replacement,
secure cookie issuance, ticket creation, a real WSS-to-PTY echo lifecycle, and
stable failure behavior. The audit scan requires the canonical synthetic
identity, configured target, start/end metadata, and actual backend peer while
excluding authorization values, cookies, tickets, terminal input/output,
commands, private keys, and fixture secrets.

Negative startup checks run the real daemon with incorrect public URL,
plaintext production exposure, development authentication, and incomplete
trusted-proxy configuration and require fail-before-listen behavior.
Untrusted-peer rejection uses a second network source outside the configured
proxy address.

## Cleanup and diagnostics

The smoke harness owns every generated name and path. One trap handles normal
exit, assertion failure, interruption, and cancellation. It stops and removes
client, proxy, auth, and ttygate containers; removes the internal network and
test-only image; deletes certificates, keys, configs, logs, audit files, and
fixtures; and verifies no labeled Docker object or recorded host process
survives. Proxy and ttygate stop paths are exercised while a PTY is live.

Failures print bounded container state and recent logs after redacting
credential-like strings. Generated diagnostics are retained only in the
current command output, never as CI artifacts. Private keys, cookies, tickets,
authorization values, and terminal content are never printed.

CI runs separate required Caddy and Nginx jobs on disposable Linux runners.
Proxy images and all added actions are pinned immutably. Chunk 4.2 is marked
complete only after both full lifecycle jobs are green; local platform skips
are reported as skips rather than successes.

## Public documentation

The README gains a concise production checklist, an authentication-provider
matrix, complete local PTY and strict-host-key SSH examples, a safe synthetic
audit JSONL record, a concise Shell In A Box comparison, and explicit
non-goals. The roadmap, rewrite plan, and threat model record the verified
deployment boundary while retaining the pre-release warning and Chunk 4.3
release gate.

The non-goals state that local PTYs share the daemon OS user; the browser
cannot supply arbitrary commands or arguments; ttygate has no native SSH,
automatic host-key learning, built-in Cloudflare/Tailscale authentication,
native ACME, release, or production-readiness claim.
