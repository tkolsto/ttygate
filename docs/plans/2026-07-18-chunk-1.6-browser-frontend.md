## Settled design

### UX and stable state model

The page has a compact header, target selector, primary connect/close action, read-only badge, `role="status"` state line, and the xterm surface. The terminal is the visual focus, fills the available viewport, retains useful output after closure, and remains readable at narrow widths. Every control has a visible label and keyboard focus treatment. The page never injects HTML from a server response.

The DOM-free state model exposes these stable phases:

- `establishing-identity`
- `ready`
- `requesting-authorization`
- `connecting`
- `active`
- `denied`
- `protocol-error`
- `internal-error`
- `timed-out`
- `exited`
- `transport-disconnected`
- `user-closed`

Only `active` accepts input or sends resize controls. `ready` is reached after identity establishment and authenticated target discovery. Terminal output remains available in all terminal states. Read-only sessions enter `active` but never attach keyboard, paste, or binary input listeners. There is no automatic reconnect or reattachment; the user must explicitly start a new connection.

### Authoritative target and read-only metadata

Two alternatives are rejected:

1. Hard-coding a target name in the browser would create an undocumented authority path and fail for other valid configurations.
2. Returning only a discovery catalog would make the UI's read-only presentation vulnerable to stale metadata at ticket issuance.

Add authenticated, exact-Origin `POST /api/targets`, returning only:

```json
{"targets":[{"name":"shell","readOnly":false}]}
```

`POST` is deliberate because the existing browser-authority middleware requires an `Origin` header and browsers consistently attach it to same-origin POSTs. Names and flags come from the immutable `TargetAllowlist` in deterministic configured-name order. Executables, argv, SSH hosts/options, credentials, identity, limits, and backend details are absent.

Extend successful `POST /api/sessions` to return:

```json
{"ticket":"<opaque>","target":{"name":"shell","readOnly":false}}
```

The metadata is derived from the exact resolved `Target` cloned into the ticket. It is presentation data, not authorization: the server still resolves the configured name during redemption and `Session` still enforces read-only input. Rust privacy/rejection tests prove both APIs expose only the documented fields and reject missing identity, wrong Origin, malformed bodies, and unknown targets safely.

### Identity, ticket, and socket flow

1. Preserve `establishIdentity()` as the first browser action.
2. Fetch the authenticated target catalog with same-origin credentials.
3. On explicit connect, request a grant from `POST /api/sessions`.
4. Construct the WebSocket URL from `location.protocol` and `location.host` with the fixed path `/api/ws`; do not accept URL input and do not add query, fragment, or subprotocol values.
5. Keep the ticket in one local mutable variable only. On socket `open`, send exactly `JSON.stringify({ticket})` as the first complete text message, then overwrite the local variable immediately.
6. Set `binaryType = "arraybuffer"`, attach terminal input only after the authentication envelope has been sent, transition to `active`, fit/focus the terminal, and send the measured initial size.

The ticket is never placed in controller state, a URL, DOM data, storage, analytics, logging, exception text, or user-visible status. Safe error classes and mappings never use response bodies, thrown error messages, close reasons, or hostile frame text. Unit tests install throwing spies for console and browser storage and inspect every URL, state snapshot, and rendered string.

### Protocol, input, output, and limits

Every post-authentication text frame uses the existing TypeScript codec:

- client resize: `encodeClientControl({type:"resize", cols, rows})`
- client close: `encodeClientControl({type:"close"})`
- server text: `decodeServerControl(text)`

Client input is sent only as binary `Uint8Array`. `chunkUtf8Input` iterates Unicode code points, encodes chunks no larger than 65,536 bytes, and never splits a UTF-8 sequence. xterm binary input is converted byte-for-byte and chunked at the same hard limit. Empty chunks are dropped. Read-only sessions suppress both input sources.

Server binary `ArrayBuffer` data passes through `decodeServerBinary` for the shared 65,536-byte bound and is written to xterm as `Uint8Array`; application code performs no JSON, HTML, string, or UTF-8 conversion. `BoundedTerminalWriter` serializes xterm writes through drain callbacks and caps pending output at 1 MiB. Exceeding the cap fails closed into `internal-error`, stops input, and closes the socket.

Resize events originate from actual xterm dimensions after `FitAddon.fit()`. A pure coalescer validates `1..=4096`, holds only the newest dimensions, debounces for 50 ms, suppresses duplicates, and clears its timer on teardown.

### Typed close/error mapping and stale-event protection

The controller retains portable exit status when received, then maps final controls:

- `close/exited` → `exited`, displaying code, signal, or unavailable status
- `close/timeout` → `timed-out`
- `close/policy` or curated authorization/policy error → `denied`
- `close/protocol-error` or malformed/unexpected server frame → `protocol-error`
- `close/internal-error` or output-cap failure → `internal-error`
- WebSocket loss without a typed terminal control → `transport-disconnected`
- explicit user action → `user-closed`

Each connect/replace/close increments a monotonically increasing generation. Every fetch continuation, socket listener, timer, xterm listener, and write-drain callback captures that generation and verifies it before changing state or sending. Earlier WebSocket events therefore cannot mutate the current session.

One idempotent teardown routine disables input first, disposes terminal listeners, clears resize/connection timers, invalidates pending output, attempts one codec-encoded close control only when orderly close is possible, closes the WebSocket, detaches all socket handlers, and drops references. `pagehide` and `beforeunload` call the same path without reconnect. Repeated connect/close cycles must leave one current handler set and zero timers from prior generations.

### Browser integration and visual QA

Add a Playwright smoke runner that builds the frontend, starts a real `ttygated` process on loopback with a temporary config pointing only at the existing PTY fixture, launches Chromium, and proves:

- identity bootstrap and authenticated target discovery;
- ticket request and first-message WebSocket authentication;
- real PTY `READY`, echo, UTF-8 paste, and resize output;
- explicit close and `user-closed`;
- natural `exit`, portable exit status, and `exited`;
- a routed authorization rejection and `denied`;
- the page URL remains the fixed origin root and contains no ticket or credentials.

The runner always terminates and awaits the daemon, records fixture leader/descendant PIDs where available, and fails if those processes remain. Browser-control QA separately inspects desktop and narrow viewports, target selection, focus, keyboard operation, terminal sizing, status announcements, read-only state, and closed output. Screenshots supplement interaction assertions rather than replace them.

| Boundary | Design check |
|---|---|
| Issue #7 / Roadmap 1.6 | Covers ticket-first connection, xterm output, resize, paste, typed states, read-only suppression, real browser smoke, and manual QA. |
| Chunk 1.3 | Preserves dev identity bootstrap, exact Origin, real cookie, configured target resolution, and bounded ticket issuance. |
| Chunk 1.5 | Uses fixed same-origin `/api/ws`, first-message ticket envelope, codec-only post-auth frames, bounded binary messages, and no reconnect. |
| Target authority | Catalog and grant metadata expose only configured name/read-only presentation; executable/argv/SSH/credentials remain server-only. |
| Read-only authority | Browser suppression is based on authenticated server metadata; server-side session enforcement remains decisive and is regression-tested. |
| Ticket secrecy | Ticket exists only in a local variable until first send, is immediately overwritten, and never reaches state/URL/DOM/storage/log/error text. |
| Hostile output | Binary reaches xterm as bytes; controls are decoded by the existing codec; UI status uses fixed text and `textContent`. |
| Backpressure | Pending xterm output is serialized and hard-capped; overflow tears down instead of growing memory. |
| Lifecycle | One generation-scoped controller owns listeners, timers, queues, socket, and orderly close; stale events are inert. |
| Future controls | Production auth, TLS/proxy gating, audit persistence, SSH execution, recording, and reconnect remain future work. |
