# Chunk 1.5 Authenticated WebSocket Bridge Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Implement Roadmap Chunk 1.5: an Origin- and cookie-authenticated `/api/ws` endpoint whose first bounded text message atomically redeems a single-use identity-bound ticket before starting one allowlisted PTY session, then bridges the existing binary/control protocol with bounded backpressure and explicit teardown.

**Architecture:** `server.rs` owns only HTTP upgrade authority checks and immutable `AppState` wiring. A new `websocket.rs` module owns the pre-protocol ticket envelope, safe error mappings, post-handshake codec adaptation, bounded reader/writer tasks, cancellation, and the one-connection/one-session lifecycle. `SessionManager` remains the only execution authority: the bridge passes only the authenticated `Identity`, the redeemed configured target name, and a fixed validated `80x24` size.

**Tech Stack:** Rust 2024, axum 0.8 WebSockets, Tokio bounded channels/watch cancellation/timeouts/tasks, futures-util split/sink/stream traits, existing ttygate auth/origin/ticket/protocol/session APIs, tokio-tungstenite loopback integration clients.

**Issue:** Refs #7. Chunk 1.6 remains required to close #7.

## Settled design

### Authority boundary and handshake

- Add only `GET /api/ws`. Reject a query string and any `Sec-WebSocket-Protocol` value; the endpoint has no path parameters and reads no client-supplied target, session id, command, credential, ticket, executable, or argv from the HTTP request.
- Apply the existing exact-one-Origin middleware before the handler. In the handler, require exactly one syntactically valid cookie header and authenticate it before calling `WebSocketUpgrade::on_upgrade`.
- Configure axum's WebSocket message limit to `protocol::MAX_BINARY_BYTES` and its frame limit no higher than that value. Axum/tungstenite reassembles fragmented frames into one bounded `Message`; application code never receives or accumulates fragments. The first application message must be one complete non-empty text message no larger than a deliberately smaller handshake bound.
- The pre-protocol envelope is exactly `{"ticket":"<opaque-ticket>"}`. A custom closed-schema deserializer rejects non-objects, duplicates, missing fields, unknown fields, wrong types, empty values, and malformed JSON without retaining or reflecting the raw payload.
- Authenticate the cookie before upgrade; after upgrade require the envelope within a short fixed deadline. Redeem through `TicketStore::redeem(ticket, identity)`, which already performs one-lock TTL, single-use, identity binding, target binding, and wrong-identity non-consumption.
- Accept only a redeemed `Target::Pty`. Pass only `target.name()` to `SessionManager::start` with `Resize::new(80, 24)`. An SSH ticket is consumed and rejected safely. Any start failure consumes the redeemed ticket and emits only a curated failure.
- A successful upgrade, ping/pong traffic, or any failed/expired/malformed handshake never starts a child.

### Bridge ownership and bounded flow

- The upgraded connection owns exactly one `Session`.
- Split the socket. A reader task owns the stream and forwards only complete typed inbound messages through a bounded Tokio channel. A writer task owns the sink and drains a bounded outbound channel.
- The connection coordinator owns `Session`, both join handles, and cancellation. It selects among inbound frames, session output/closure, reader completion, writer completion, and cancellation.
- Every bounded send and every socket write is selected against cancellation. Session output pauses when the outbound channel is full, which propagates through `Session::read` into Chunk 1.4's bounded PTY channel.
- Binary client messages go through `protocol::decode_client_binary` and then `Session::write`. Text client messages go through `protocol::decode_client_control`; resize calls `Session::resize`, close calls `Session::close`. Ping/pong never become terminal data. Peer close or transport loss triggers teardown.
- Session output is sent as `Message::Binary` without UTF-8, JSON, HTML, or logging interpretation.
- On natural terminal state, convert the typed outcome/reason to existing encoded server controls. Send portable exit status only for `ChildExited`, then the final close control, then the WebSocket close handshake.
- On malformed protocol input, attempt one generic `protocol-error` control and close with 1008. On application-observed oversize, close with 1009; transport-observed oversize is delegated to tungstenite's configured limit.
- On every exit, cancel both paths, stop input, explicitly `Session::close` or `wait_closed` as appropriate, await process-group teardown, close the outbound queue, and join both tasks. Task drop and child kill-on-drop are defense in depth, not the lifecycle mechanism.

### Safe mapping

- `TicketError::{Malformed,Unknown,Expired,WrongIdentity}` maps to one non-reflecting `authorization-denied` message and policy close.
- `SessionError::{GlobalLimit,IdentityLimit,TargetUnavailable,ManagerClosed,ReadOnly}` maps to stable policy/session errors; spawn/backend/closed/invalid-transition failures map to one stable unavailable/internal error. No `Display` string is sent.
- `SessionCloseReason::ChildExited` maps to optional portable exit status followed by `CloseReason::Exited`.
- `Explicit` maps to `ClientRequest`; idle/absolute timeout maps to `Timeout`; manager shutdown maps to `Policy`; handle drop maps to `TransportError`; backend failure maps to a curated unavailable error plus `InternalError`.
- Protocol codec failures map only to the generic protocol error and 1008, except size failures which use 1009.

## Critical design review

| Contract | Design check |
|---|---|
| ADR 0002 | Split socket, bounded input/output channels, cancellation-selectable sends/writes, owned joins, explicit session teardown. |
| ADR 0003 | Bridge never sees backend handles; `SessionManager` continues to own spawn and mandatory process-group reap. |
| Roadmap 1.5 | Ticket is first message, real PTY echo/resize tests, wrong Origin and dropped-socket teardown are in scope. |
| Roadmap 1.6 | No frontend WebSocket, ticket fetching, xterm.js, paste, or UI state changes. |
| Rewrite plan | No ticket in URL, no bare WebSocket, one-time identity binding, target allowlist provenance, no reattachment. |
| Threat model | Cookie and Origin precede upgrade; ticket precedes child; hostile bytes/errors are not reflected or logged; buffers are bounded. |
| Protocol | Existing codec owns all post-handshake binary/text messages and 4,096/65,536-byte limits; typed terminal controls remain unchanged. |
| Issue #7 | Produces the backend prerequisite only and uses `Refs #7`; does not claim frontend acceptance criteria. |

Risks found during review and their required tests:

1. axum exposes reassembled WebSocket messages, not raw continuation frames. Set both transport frame/message bounds, keep the handshake bound below both, and test a fragmented client message is either delivered only as one complete bounded message or safely rejectedâ€”never parsed incrementally.
2. Concurrent socket failure while the coordinator waits on a full outbound channel can deadlock without independent cancellation. Add a capacity-one unit seam and a real output-flood/drop regression.
3. `Session::read()` and `wait_closed()` both require mutable access. The coordinator must await `read()` until its channel closes, then call `wait_closed()` sequentially; do not spawn detached session borrowers.
4. A client close request must not let the reader continue feeding input. Cancel the input path before awaiting `Session::close`.
5. A redeemed SSH ticket or PTY spawn failure is intentionally consumed. Assert a second connection cannot reuse it.
6. Existing `AppState::new` can accidentally accept mismatched ticket and session allowlists if wiring remains independent. Construct `SessionManager` inside `AppState` from the same cloned immutable `TargetAllowlist` and `Limits`.
