# Chunk 4.1 Docker and systemd Design

## Scope

Chunk 4.1 packages the merged M1-M3 daemon for two Linux execution paths:
a reproducible container image and a hardened systemd service. It does not add
reverse-proxy examples, production deployment guidance, release automation,
artifacts, images, or a v0.1 tag. Those remain in Chunks 4.2 and 4.3.

The package defaults are deliberately useful only on the local host. Both
configurations bind `127.0.0.1:7681`, run development authentication, write a
required JSONL audit log, and expose one allowlisted `/bin/sh` PTY target. An
operator must supply a different validated configuration to use direct TLS,
trusted-proxy production authentication, or SSH targets. Packaging never
relaxes the daemon's existing fail-before-bind configuration, audit, TLS, or
SSH checks.

## Docker design

The Dockerfile uses three digest-pinned stages. A Node stage performs `npm ci`
and rebuilds the frontend from the lockfile. A Rust stage builds the daemon
with `Cargo.lock` and `--locked` after copying those generated assets. A Debian
slim runtime stage contains only the stripped daemon, the OpenSSH client needed
by configured SSH targets, the minimal shell used by the example target, CA
certificates, and their shared-library/runtime dependencies. Build tools,
package-manager indexes, caches, repository source, tests, fixture keys, and
the Node/Rust toolchains do not cross into the runtime stage.

The runtime identity is a fixed, dedicated `ttygate` UID/GID so bind-mounted
audit and SSH material can be provisioned with stable ownership. The config
under `/etc/ttygate` is root-owned and read-only. `/var/log/ttygate` is the
only image-created service-owned persistent writable path. Static assets are
embedded in the binary at compile time; there is no separate writable asset
tree. SSH identities and known-host files are operator mounts under
`/etc/ttygate/ssh`; their ownership and modes must still pass the daemon's
existing checks.

The image declares no public-facing default. Its configuration binds only the
container loopback interface, and the smoke test proves that a published host
port cannot reach the daemon. The daemon gains a bounded, non-reflecting
`--health-check` command that performs an HTTP request to the configured
loopback health address and accepts only the exact `/healthz` success response.
Docker `HEALTHCHECK` invokes that command without a shell. The daemon remains
PID 1, so Docker signals it directly; the smoke suite verifies stop and forced
failure cleanup and removes its containers, networks, images, and fixtures on
every exit path.

## systemd design

The service uses a dedicated static `ttygate` account rather than
`DynamicUser`. Stable ownership is required because the daemon deliberately
rejects SSH identities, known-host files, and existing audit files not owned by
its effective user. The accompanying sysusers and tmpfiles files create the
account and service-owned state/log directories while keeping
`/etc/ttygate/ttygate.toml` root-owned and non-writable by the daemon.

The unit uses `Type=notify`, `NotifyAccess=main`, and `WatchdogSec`. After the
listener has bound successfully, the daemon sends `READY=1`. When systemd
enables its watchdog, an event-loop task sends `WATCHDOG=1` at half the
negotiated interval. A blocked runtime therefore stops notifying and systemd
restarts the cgroup. Docker health remains tied directly to `/healthz`; systemd
watchdog behavior is exercised through `sd_notify`.

The unit sets `NoNewPrivileges`, an empty capability and ambient-capability
set, `ProtectSystem=strict`, protected home/kernel/control-group/clock/hostname
surfaces, private temporary files and devices, restricted address families,
namespaces, realtime, SUID/SGID, personality changes, writable executable
memory, and a syscall allowlist appropriate to a network service that must
open PTYs and spawn allowlisted children. `KillMode=control-group` and bounded
stop time make systemd own last-resort cleanup for the daemon, OpenSSH clients,
PTY leaders, and descendants.

`PrivateNetwork` cannot be enabled because network access is the service's
purpose. `PrivateUsers` is intentionally omitted: UID remapping would conflict
with the fail-closed ownership checks for audit and SSH material and would make
stable host provisioning misleading. `PrivateDevices` remains enabled because
systemd's private device set includes the pseudoterminal devices the daemon
requires. These exceptions and their executable tests are explicit rather than
silent profile weakening.

## Verification and diagnostics

`scripts/check-packaging.sh` provides fast, platform-independent assertions for
digest pinning, build-context exclusions, non-root identity, exact runtime
paths, Docker health configuration, the localhost package defaults, and every
required systemd hardening directive. It fails with labels and file locations,
never file contents or secrets.

`scripts/smoke-docker.sh` builds twice from the clean repository input, compares
the resulting configuration/rootfs identity, inspects the runtime filesystem
and effective user, starts the minimal configuration, observes Docker health,
proves loopback-only behavior, exercises normal stop and failure cleanup, and
uses traps to remove all generated objects.

`scripts/smoke-systemd.sh` runs only on a supported Linux systemd environment.
It verifies the unit, installs disposable files into an isolated test root or
ephemeral nested systemd environment, exercises readiness/watchdog/start/stop/
failure/restart behavior, inspects effective hardening, and checks that no
daemon, SSH, PTY child, unit, or fixture survives cleanup. CI supplies the
supported Linux environment because the development host is macOS.

Smoke failures print bounded status, unit/container state, and recent
non-secret service logs while preserving no secret fixtures. CI adds dedicated
Docker and systemd packaging jobs, and the public documentation checker asserts
that Chunk 4.1 status is reconciled without claiming the rest of M4 complete.
