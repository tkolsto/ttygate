# Chunk 0.3 — Decision Spikes Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Resolve the web-stack, PTY, and OpenSSH subprocess architecture questions with three disposable programs, reproducible experiments, and three evidence-backed decision records for GitHub issue #3.

**Architecture:** Keep all non-production experiment code under `spikes/`, outside the Cargo workspace, with one command per spike that proves the acceptance behaviors and exits non-zero on failure. Commit only deterministic fixtures, scripts, lockfiles, summarized evidence, and decision records; disposable containers, keys, known-hosts files, logs, and build output stay in temporary directories or ignored targets.

**Tech Stack:** Rust stable; axum/tokio/tokio-tungstenite; portable-pty, pty-process, rustix/nix comparisons; system OpenSSH client; Docker-hosted disposable OpenSSH server; shell acceptance runner.

## Global constraints and evidence rules

- Work only in `.worktrees/chunk-0.3-decision-spikes` on `feat/chunk-0.3-decision-spikes`, based on `main`; never create or use `master` and never modify the former PR #13 branch/worktree.
- Spike code is evidence, not production code. It must not be added to the root Cargo workspace or imported by `ttygated`.
- No Shell In A Box source or GPL-derived expression enters the repository.
- All subprocess argument vectors are arrays constructed by spike code. No shell command string, browser value, or client-controlled SSH option is accepted.
- All SSH experiments use `StrictHostKeyChecking=yes`, an explicit temporary `UserKnownHostsFile`, and `BatchMode=yes` for key-authenticated noninteractive probes.
- Each experiment writes only sanitized, reproducible evidence. Never commit generated private keys, container credentials, host keys, absolute temporary paths, or terminal transcripts containing secrets.
- Dynamic maintenance claims must name the checked date and primary source (crates.io metadata, repository releases/commits/issues, crate source, or official documentation). Record observations, not unsupported popularity claims.
- Tests for retained spike harness behavior follow red → green. Exploratory crate API probes are explicitly throwaway under the TDD exception, then exercised by a failing acceptance command before being considered evidence.
- Decision records use sequential identifiers `0002`–`0004`, status `accepted`, and sections: Context, Decision, Alternatives, Evidence, Risks and mitigations, Consequences.
- Commits reference issue #3 where useful and use conventional messages with no AI attribution.

## Critical pre-execution review

- **Roadmap alignment:** The plan produces exactly three disposable programs plus three records and leaves production crates unchanged. It closes both Phase 1 blockers (web and PTY) and records the Phase 3 SSH contract without pulling Phase 1/3 implementation forward.
- **Security alignment:** Negative paths cover unknown/mismatched hosts, refused connections, client-option exclusion, bounded bridging, disconnect teardown, process reaping, and descendant absence. Keys and host files are ephemeral and server host identity is sourced out-of-band from the disposable fixture.
- **Evidence quality:** Every choice requires live behavior plus dated primary-source maintenance data. Repetition catches teardown races, while sanitized Markdown summaries keep results reviewable without committing volatile logs.
- **TDD fit:** Retained harness and argv policy begin RED. The programs are intentionally throwaway prototypes, the skill’s explicit TDD exception; nevertheless, every claimed behavior must first fail at the acceptance boundary and then pass.
- **Portability risk:** Development runs on macOS while production primarily targets Linux. The direct PTY comparison must distinguish observed macOS results from Linux API/source evidence, and no record may imply untested cross-platform behavior.
- **Disposable SSH risk:** Docker availability and image acquisition are external prerequisites. The runner must fail clearly, clean up unconditionally, bind only loopback, and never silently replace a real-server test with a mock.
- **Decision threshold:** A candidate is not selected merely because it compiles. The chosen stack must satisfy lifecycle ownership, resize, bounded async integration, error classification, and testability with no unresolved question blocking Chunk 1.4/1.5.
