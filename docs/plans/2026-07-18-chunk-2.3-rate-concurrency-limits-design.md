# Chunk 2.3 Rate and Concurrency Limits Design

**Issue:** Refs #24. The final pull request may use `Closes #24` only after
every acceptance criterion is verified.

**Scope:** Implement the smallest complete Roadmap Chunk 2.3 without changing
the existing Origin, trusted-peer, cookie, target, ticket, WebSocket, PTY,
transport, or process-authority boundaries.

## Selected architecture and alternatives

The selected design has two independent in-memory rate limiters and one shared
capacity authority:

1. a session-request limiter keyed by authenticated `Identity`;
2. an authentication-failure limiter keyed only by the listener-supplied peer
   IP address, with one closed missing-peer key; and
3. the session manager's global/per-identity capacity authority, extended to
   issue move-only pending leases before tickets are created and to activate a
   lease exactly once when its ticket is redeemed.

Each rate limiter uses a first-request-anchored fixed window. This gives
constant storage per key, deterministic monotonic-time tests, and easily
audited boundary behavior. A sliding timestamp log was rejected because its
memory grows with both the key count and configured allowance. A token bucket
was rejected because fractional refill and rounding make the public boundary
contract harder to explain and test. A global-only limiter was rejected
because one identity or shared proxy could deny service to every other user.

Concurrency stays in the session manager instead of adding a second counter to
the ticket store. Tickets own pending manager leases, and successful redemption
moves the same lease into the live supervisor. This prevents disagreement,
double accounting, and transient over-limit windows.

## Configuration and defaults

The existing `[limits]` table gains four optional fields:

```toml
session_requests_per_window = 10
session_request_window_seconds = 60
authentication_failures_per_window = 20
authentication_failure_window_seconds = 60
```

Missing fields use those defaults in development, direct-TLS production, and
trusted-proxy production. Explicit values must be nonzero and fit the typed
configuration bounds; window seconds must convert to a nonzero `Duration`.
Unknown fields continue to fail through the closed TOML schema. Errors name
only the configured field and a static rule, never a supplied identity,
address, header, cookie, ticket, target, or parser diagnostic.

Restart creates empty limiter and capacity state. A restart with lower limits
therefore applies the new limits immediately rather than importing stale
in-memory counters. No state is persisted in this chunk.

## Fixed-window rate semantics

A key's first admitted operation starts its window at monotonic time `t`.
Exactly the configured number of operations are admitted while
`now < t + window`. The next operation is rejected. At exactly
`now == t + window`, the old bucket is stale, a new window starts, and the
operation is admitted. Rejected operations do not consume allowance and do not
extend the window.

The configured allowance is also the instantaneous burst size. Because windows
are anchored per key, up to twice the allowance can occur across an exact
window boundary. That is an intentional fixed-window tradeoff and is documented
rather than described as a smooth rate.

Limiter operations are serialized under one non-poisoning mutex, so concurrent
requests have exactly the configured number of winners. Buckets contain only a
key, start instant, generation, and count. Stale buckets are evicted on access.
Each limiter is hard-capped at 4,096 live keys. If the cap is full and no stale
bucket can be removed, a new key fails closed with the same public exhaustion
response; it is not inserted. Thus hostile unique peers cannot grow memory
without bound, although they can create a documented temporary denial of
service until a window expires.

An authentication attempt provisionally consumes one failure slot before
calling the provider. Successful authentication rolls that slot back within
the same bucket generation; failure commits it. This prevents concurrent
failures from all passing a separate pre-check while ensuring success does not
consume failure allowance. Once a peer exhausts failures, even a subsequently
valid authentication attempt from that peer waits for the window boundary.

## Key provenance and request ordering

Authenticated `POST /api/sessions` calls are keyed by the exact case-sensitive
`Identity` returned by the configured provider. Identity rather than peer is
used so unrelated users behind one trusted reverse proxy or NAT do not share a
session-request budget; distributing one stolen identity across peers still
shares one budget. Every authenticated call consumes session-request allowance
before body, target, or ticket processing, including malformed or unknown
targets. A rate-rejected call consumes nothing.

Authentication failures on `/api/identity`, `/api/targets`,
`POST /api/sessions`, and the WebSocket upgrade are keyed by
`ConnectInfo<SocketAddr>::ip()`. Ports are ignored. IPv4, IPv6, and
IPv4-mapped IPv6 retain their listener-reported address family, matching Chunk
2.2. Missing listener provenance uses one internal closed key and cannot be
split using client input.

`Forwarded`, `X-Forwarded-For`, `X-Real-IP`, Host, query parameters, cookies,
tickets, configured identity headers, and WebSocket subprotocols never select
either limiter key. Exact Origin enforcement remains middleware-first; an
Origin denial reaches neither authentication nor either limiter. Trusted-proxy
peer validation remains inside the provider and therefore an untrusted actual
peer consumes only its own actual-peer failure allowance.

## HTTP errors and diagnostics

Rate exhaustion uses HTTP 429 with stable JSON:

- `authentication-rate-limited` — `Authentication is temporarily limited.`
- `session-rate-limited` — `Session requests are temporarily limited.`

The response includes an integer `Retry-After` equal to the ceiling of the
remaining window in seconds, with a minimum of one. Storage-cap exhaustion uses
the full configured window as the retry interval. Concurrency exhaustion uses
HTTP 503 with distinct static codes:

- `global-session-limit` — `Global session capacity is unavailable.`
- `identity-session-limit` — `Identity session capacity is unavailable.`

No error or future audit event may contain limiter keys, identity text, peer
text, forwarded headers, cookie values, tickets, targets, or internal counter
state. Future audit integration may record only the stable denial category and
already-approved lifecycle metadata.

## Ticket-time concurrency lifecycle

The capacity authority stores at most `max_sessions` lease records under one
mutex. Each record contains an opaque internal lease id, canonical identity,
and state:

- `Pending { expires_at }` while an opaque browser ticket owns it; or
- `Live` after successful ticket redemption transfers it into a supervisor.

At `POST /api/sessions`, the server first purges monotonic-expired pending
leases, then atomically checks global and per-identity counts and inserts one
pending lease with the ticket TTL. Only after that succeeds does it generate
and insert the ticket. Ticket-generation or store-capacity failure drops the
lease immediately. Concurrent callers serialize at the capacity mutex and
have exactly the configured number of winners.

The ticket entry owns the move-only pending lease alongside the existing
identity, target, and expiry. Correct-identity redemption under the ticket
mutex removes the entry and returns one grant containing the target and lease.
Wrong identity leaves both untouched. Malformed, unknown, and replayed tickets
cannot address or release any lease. At the exact expiry boundary, redemption
removes the entry and drops its lease. Ordinary ticket-store access evicts
other expired entries; capacity admission also treats expired pending leases
as absent, so an abandoned ticket cannot block the next eligible request even
if no cleanup task ran at the boundary.

The WebSocket path consumes the grant and asks the session manager to activate
that same lease. Activation atomically changes `Pending` to `Live`; it never
increments counts. Target rejection, manager shutdown, activation failure,
PTY spawn failure, task cancellation, or panic drops the grant/lease exactly
once. Once the supervisor owns it, natural exit, explicit close, timeout,
dropped transport, backend failure, panic, and shutdown release it only after
the existing bounded process-group teardown completes.

Manager shutdown closes admission, invalidates all pending leases, requests
closure of live supervisors, and awaits their teardown. A ticket holding an
invalidated lease cannot later become live. RAII release is idempotent when a
lease was already expired or invalidated, preventing underflow and
double-release.

## WebSocket failure and ticket behavior

A failed upgrade or transport drop before first-message redemption does not
consume the ticket. Its reservation remains valid only until the original TTL,
matching current ticket authority. A syntactically invalid handshake cannot
release another ticket. Correct redemption consumes the ticket exactly once;
any later WebSocket or PTY failure releases the transferred lease rather than
restoring the ticket. A different identity's redemption attempt does not
consume the ticket or alter capacity.

No PTY starts when rate or ticket-time concurrency rejects a request. The
existing session-manager check remains as defense in depth for direct internal
callers, but the browser flow transfers rather than duplicates its reservation.

## Testing and retained evidence

Unit tests use injected monotonic clocks for exact window and TTL boundaries.
Concurrency tests synchronize real threads/tasks and assert exact winner
counts, bounded key/lease state, no underflow, and recovery. HTTP and real
development/direct-TLS/trusted-proxy integrations exercise actual Origin,
listener peer, cookie, rate, ticket, WSS, PTY, and teardown paths rather than
injecting pre-authorized identities or permits.

Cleanup fixtures retain bounded shutdown, panic cleanup, process-group probes,
and final process inspection. Race-sensitive issuance, expiry, wrong-identity
redemption, WebSocket failure/drop, spawn failure, and shutdown tests run in at
least 20 independent processes before publication.

## Documentation and deferred scope

README, roadmap, rewrite plan, threat model, public-document assertions, and
retained evidence will describe only verified behavior. Fixed-window limits
reduce single-key abuse; they do not prevent distributed denial of service,
multi-identity floods, upstream proxy exhaustion, kernel/socket exhaustion
before the application runs, or resource exhaustion inside an allowed child.
Operators remain responsible for trusted-proxy authentication, header
stripping/injection, exclusive backend reachability, and upstream network
controls.

Persistent audit storage, SSH, recording, reconnect, packaging, deployment
examples, release automation, and release hardening remain later chunks.
